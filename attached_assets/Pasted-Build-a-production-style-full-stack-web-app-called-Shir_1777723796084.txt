Build a production-style full-stack web app called “ShiroScan”.

High-level goal:
ShiroScan is a Japan-first utility AI product that helps users scan Japanese packaged food labels and quickly understand whether the product is safe for their allergies or diet preferences.

This version should be stateless and should NOT use any database, authentication, or Supabase.
Do not add login, user accounts, saved history, cloud storage, or persistent profiles.
All analysis should happen per request.
The app should feel like a reliable consumer utility, not a generic chatbot.

Primary user problem:
Japanese food labels are often dense, small, and hard to interpret quickly.
Users need a fast, trustworthy answer with visible proof, especially for allergy, ingredient, and dietary concerns.

Target users:
- Japanese consumers who want easier label understanding
- Foreign residents in Japan
- Travelers in Japan
- People with food allergies
- Users with dietary preferences like vegetarian, halal, no pork, lactose-free

Non-goals for MVP:
- Do not build a general AI assistant
- Do not add database or auth
- Do not support all product categories initially
- Do not make medical claims
- Do not overuse agentic workflows or complex orchestration in v1
- Keep v1 focused on packaged food labels only

Tech stack:
Frontend:
- Next.js 14+ with TypeScript
- App Router
- Tailwind CSS
- shadcn/ui or similarly clean component approach
- Mobile-first responsive design

Backend:
- Python FastAPI service for OCR parsing, label analysis, and AI workflow orchestration
- Separate Python service folder with modular architecture
- Pydantic models for request/response schemas
- Async-ready backend code
- Clear service boundaries between OCR, parsing, rule engine, and LLM summarization

AI / model usage:
- Use Gemini models for vision-related tasks where useful, such as extracting text or understanding label images
- Use Groq-hosted LLMs for fast text reasoning, bilingual summarization, and explanation generation
- Build provider abstraction so models can be swapped later
- Support mock/fallback mode when API keys are not present

Architecture requirement:
This must be built like a real product, not a monolithic demo app.
Use a clean layered architecture.

Frontend app:
- app/
- components/
- lib/
- hooks/
- types/
- services/

Python backend:
- backend/app/main.py
- backend/app/api/
- backend/app/core/
- backend/app/services/
- backend/app/models/
- backend/app/schemas/
- backend/app/utils/
- backend/tests/

Use a gateway pattern:
- frontend calls backend API
- backend handles OCR + analysis + explanation
- backend returns structured result to frontend
- no database writes
- no persistent storage
- file handling should be temporary only

Required product behavior:
The user uploads or takes a photo of a Japanese packaged food label.
The app:
1. validates the file
2. temporarily processes the image
3. extracts text from the image
4. identifies allergens and dietary conflicts
5. interprets important Japanese phrases
6. generates a result:
   - Safe
   - Caution
   - Avoid
7. explains why in English and Japanese
8. shows extracted proof text and confidence
9. returns the result without saving user data permanently

Analysis pipeline requirements:
Build the backend pipeline in this order:

Step 1: Image intake
- validate image type and size
- basic quality checks if possible
- use temporary file or in-memory file handling only
- clean up temp files after processing

Step 2: OCR / extraction
- Build OCR provider abstraction
- Support:
  - mock provider
  - Gemini-based extraction provider
- Normalize extracted output into a standard schema

Step 3: Rule engine
- Implement deterministic matching first
- Detect known allergen keywords and caution phrases
- Detect dietary conflicts using a configurable keyword ruleset

Step 4: Optional LLM summarization
- Use Groq LLM to convert structured findings into short user-facing explanations
- Must not invent unsupported claims
- Summaries must cite extracted evidence internally in the generated object
- If LLM fails, fallback to template-based explanation

Step 5: Decision engine
- Return Safe / Caution / Avoid using explicit rule logic
- Use confidence and evidence thresholds
- If evidence is weak, prefer Caution over overconfident Safe

API design:
Frontend should call backend REST API.
Create endpoints like:
- POST /api/analyze
- GET /api/health
- GET /api/demo-samples

POST /api/analyze should accept:
- image file
- current session preferences as JSON payload or form fields

Response from POST /api/analyze should be typed and structured like:
{
  "product_name": "string | null",
  "status": "safe | caution | avoid",
  "confidence_score": 0.0,
  "matched_allergens": [],
  "matched_diet_conflicts": [],
  "extracted_keywords": [],
  "evidence": [
    {
      "japanese_text": "...",
      "normalized_meaning": "...",
      "category": "allergen | caution | ingredient | diet_conflict"
    }
  ],
  "summary_ja": "...",
  "summary_en": "...",
  "raw_ocr_text": "...",
  "warnings": [],
  "processing_mode": "mock | gemini | fallback"
}
